# 기술 스택별 작업 규칙 (tech_stack.md)

이 가이드는 프로젝트의 언어 및 개발 환경별 코딩 규격과 엔지니어링 표준을 규정합니다. 에이전트는 코드 편집 작업을 수행할 때 본 룰을 준수해야 합니다.

---

## 1. 🐍 Python Rules

Python 코드를 작성하고 리팩터링할 때 지켜야 할 규칙입니다.
* **버전 및 코딩 표준**: Python 3.11+ 문법의 새로운 기능을 적극 활용하고, PEP 8 스타일 가이드를 준수합니다.
* **엄격한 타입 힌팅 (Strict Typing)**: 모든 함수 및 메서드 정의부에는 매개변수와 반환값의 Type Hinting을 필수적으로 적용합니다. `Any` 타입 사용은 강력히 지양하고 구체적인 타입을 선언합니다.
* **의존성 관리 (`uv`)**:
  - 패키지 의존성 관리에 최신 고속 도구인 `uv`를 사용합니다.
  - 패키지를 추가하거나 버전을 업데이트할 때는 관련 도구를 실행하여 `pyproject.toml` 및 Lock 파일을 갱신해야 합니다.
  - 구식 `requirements.txt` 파일은 관리 범위에서 배제하며 변경/수정 시도하지 않습니다.

---

## 2. 🔷 TypeScript & Vue 3 (Frontend) Rules

프런트엔드 애플리케이션 개발 시 지켜야 할 규칙입니다.
* **Vue 3 컴포넌트 개발**:
  - 컴포넌트 개발 시 반드시 `<script setup>` 문법과 Composition API를 사용합니다.
  - Options API 등 레거시 문법 작성을 금지합니다.
* **타입 안전성 (No Any)**:
  - TypeScript의 `any` 타입 사용을 엄격히 금지합니다.
  - 복잡한 데이터 모델이나 API 요청/응답 스키마는 인터페이스(`interface`)나 타입(`type`)을 명확하고 구체적으로 선언하여 관리합니다.
* **스타일 가이드 및 포맷터**:
  - 프로젝트 루트 디렉토리에 정의된 `.eslintrc` 및 `.prettierrc` 설정을 에이전트 구동 환경이 자동 감지하도록 포맷팅 설정을 유지하고 준수합니다.

---

## ⚙️ 공통 엔지니어링 & 아키텍처 규칙

* **Strict OOP Patterns**: 클래스, 인터페이스, SOLID 원칙을 우선하여 설계합니다. 핵심 비즈니스 로직에 결합도가 높은 느슨한 유틸리티 전역 함수 선언을 지양합니다.
* **Robust Error Handling**: 빈 `catch` 블록(Silent Error) 작성을 금지합니다. 예외 상황 발생 시 항상 에러 콘텍스트 정보를 로깅하고, `finally` 블록에서 데이터베이스 및 Redis 연결 자원을 명시적으로 닫아(close/quit) 자원 누수를 방지합니다.
* **Centralized Config**: `process.env`나 글로벌 설정 객체에 직접 난립하여 접근하지 마십시오. 전용 설정 로더 클래스를 통해 환경변수를 안전하게 주입받아 사용하도록 합니다.
* **Agent-Friendly Docstrings**: 모든 핵심 모듈 및 스크립트 상단에는 설계 컨텍스트, 제약 조건, 의존 관계를 명시하는 docstring 주석 헤더를 추가해 두어, 에이전트 간 리팩터링 루프가 꼬이는 일을 예방합니다.
* **No Superficial Patches**: 오류 발생 시 예외 케이스만 제외하는 임시 정규식 추가 등의 임시방편(하드코딩 패치)은 금지됩니다. 오류의 근본 흐름을 분석하여 아키텍처 관점에서 견고한 해결책을 냅니다.
