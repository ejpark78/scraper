아카이브 디렉토리를 갱신할 때 아카이브 표만 그리고 끝내던 것에서, 각 `.archive.md` 내부의 개별 피처 헤더를 grep 파싱하여 앵커 링크 리스트로 시각화 출력하도록 개편했습니다.

#### 변경 전 (Before)
```bash
        for ind in "${individual_files[@]}"; do
            local bname=$(basename "$ind")
            local prefix=$(echo "$bname" | sed -E 's/^([0-9]+)-.*/\1/')
            local desc=$(echo "$bname" | sed -E 's/^[0-9]+-(.*)\.md$/\1/')

        done
    } > "$index_file"
```

#### 변경 후 (After)
```bash
        for ind in "${individual_files[@]}"; do
            local bname=$(basename "$ind")
            local prefix=$(echo "$bname" | sed -E 's/^([0-9]+)-.*/\1/')
            local desc=$(echo "$bname" | sed -E 's/^[0-9]+-(.*)\.md$/\1/')
            echo "|   | $desc | [$bname]($bname) | 1 |"
        done

        echo ""
        echo "## 📂 상세 아카이브 내역 (Archived Details)"
        echo ""

        for arch in "${archives[@]}"; do
            local bname=$(basename "$arch")
            local range=${bname%.archive.md}
            echo "### 📦 아카이브: $range ($bname)"
            echo ""
            
            while IFS= read -r header; do
                if [ -n "$header" ]; then
                    local clean_h=$(echo "$header" | sed -E 's/^## //')
                    # Make anchor slug compatible with markdown spec
                    local slug=$(echo "$clean_h" | tr '[:upper:]' '[:lower:]' | sed -E 's/[^a-z0-9 _-]//g' | tr ' ' '-')
                    echo "- [$clean_h]($bname#$slug)"
                fi
            done < <(grep '^## [0-9][0-9][0-9]-' "$arch" || true)
            echo ""
        done
    } > "$index_file"
```

## 077-fix-archive-overwrite-issue.plan

# 📋 작업 계획서: 아카이브 덮어쓰기 문제 해결 및 Git 탐색 덤프 루프 방지 규칙 추가 (077-fix-archive-overwrite-issue.plan.md)

이 문서는 `make agents-squash` 아티팩트 아카이빙 시 기존 아카이브 파일이 덮어써져 데이터가 유실되는 문제를 해결하기 위해 스크립트를 개선하고, Git 히스토리 탐색 덤프 루프 방지 규칙을 Git Flow 규칙 및 프로젝트 AGENTS.md에 보완하는 계획을 기록합니다.

---

## 1. 🔍 문제 상황 및 필요성
- `squash-artifacts.sh`가 아티팩트를 아카이빙할 때 단순 덮어쓰기(`>`) 방식으로 작성되어 있어, 이전 턴에 squash가 진행되어 이미 원본 파일들이 지워진 상태에서 추가 squash를 실행하면 기존 아카이브 내역이 유실되는 현상이 발생했습니다.
- 에이전트가 Git의 파일 삭제 및 복구 이력을 찾는 과정에서 한 번에 좁은 범위를 파악하지 못하고 여러 턴에 걸쳐 `git log`, `git show`, `git reflog`를 단발적으로 반복 조회하는 비효율성(덤프 루프)이 확인되어 제약 규칙화가 필요합니다.

---

## 2. 🛠️ 해결 방안

### 개선 1: `squash-artifacts.sh` 아카이브 증분 추가 방식 도입
- `run_archive()` 함수 내에서 대상 `$archive_file`이 이미 물리적으로 존재하는지 체크합니다.
- 파일이 없을 경우 헤더를 초기화 생성(`>`)합니다.
- 아카이빙 대상 소스 파일들이 이미 아카이브에 기입되어 있는지 `grep -q "^## $stem"`으로 검사하여 중복을 우회합니다.
- 존재하지 않는 아티팩트 본문만 증분 병합(`>>`)한 뒤 원본을 삭제합니다.

### 개선 2: Git Flow 규칙 및 AGENTS.md에 Git 탐색 덤프 루프 방지 명시
- `.agents/rules/git_flow.md` 파일에 "Git 히스토리 탐색 덤프 루프 방지" 조항을 추가합니다.
- 프로젝트 루트 `AGENTS.md`의 규칙 16번(Git Flow 브랜치 전략 및 에이전트 행동 지침)에 덤프 루프를 돌지 말고 단일 턴에 통합 조회해야 한다는 문장을 보완합니다.

```markdown
이 문서는 프로젝트 내의 모든 코드 구현 및 설계 시 준수해야 하는 공통 아키텍처 규칙을 정의합니다.

## 1. DRY (Don't Repeat Yourself) 규칙
* **단일 원천 정보 (Single Source of Truth)**: 시스템 내의 모든 비즈니스 로직, 데이터 형태, 환경 설정 등의 '지식'은 오직 한 곳에만 존재해야 합니다.
* **로직 중복 금지**: 2회 이상 반복되거나 재사용될 가능성이 높은 공통 기능은 독립된 함수, 클래스, 혹은 공통 모듈로 분리 및 추상화하여 사용합니다.
* **설정 및 환경의 하드코딩 금지**: 포트, 호스트, 비밀번호 등은 절대 코드 내에 중복 기재하지 않고 전용 설정 관리 모듈을 통해 주입받습니다.
* **우연한 일치의 경계**: 모양은 같으나 목적과 비즈니스 맥락이 완전히 다른 코드를 무리하게 공통화하여 결합도를 높이지 않도록 주의합니다.

## 2. 기존 아키텍처 규칙 이관
* **Strict OOP Patterns**: 클래스, 인터페이스, SOLID 원칙을 사용합니다. 핵심 로직에 느슨한 유틸리티 함수 사용을 지양합니다.
* **Strict Typing**: 'any' 타입 사용을 금지합니다. public 메서드에 명시적 반환 타입과 인터페이스를 선언합니다.
* **Robust Error Handling**: 빈 catch 블록을 사용하지 마세요. 항상 에러 컨텍스트를 로깅하고 finally 블록에서 DB/Redis 연결을 종료합니다.
* **Centralized Config**: 'process.env'는 전용 설정 파일에서만 접근합니다. 생성자를 통해 설정을 주입합니다.
* **Agent-Friendly Docstrings**: 모든 소스, 스크립트, 자동화 파일에 설계 컨텍스트, 제약 조건, 의존성을 설명하는 헤더 docstring/주석을 추가하여 리팩터링 루프를 방지합니다. 동작 변경 시 업데이트합니다.
* **No Superficial Patches**: 오류 발생 시 표면적 패치(예: 커스텀 regex 제외나 하드코딩 파라미터로 증상 숨기기)를 절대 구현하지 마세요. 항상 데이터 흐름을 추적하고, DB/상태 조정을 조사하여 진정한 근본 원인을 찾아 견고한 구조적/아키텍처 솔루션을 구현하세요. 또한 버그가 수정(Bugfix)되었을 때에는 단순 변경사항과 엄격히 구분하여 CHANGELOG와 코드 리뷰 문서에 'Bugfix'임을 명확히 표기하고 기록해야 합니다.
```

| **세션 진입 시 행동 수칙** | `main` 브랜치인 경우에만 사용자에게 경고하고 `develop`으로 이동을 제안함 | `main` 및 `develop` 브랜치인 경우 모두, 실제 작업을 시작하기 전에 `git checkout -b feature/###-<name>` 명령을 생성하여 명시적 승인을 받도록 의무화 |

## 🔍 변경점 요약 (`.agents/rules/git_flow.md`)
```diff
- 0. **세션 시작 시 브랜치 자가 진단**:
-    - 에이전트는 작업을 시작할 때 항상 현재 브랜치가 `main` 브랜치인지 파악합니다.
-    - `main` 브랜일 경우 코드 수정을 절대 진행하지 않고 사용자에게 경고한 후 `git checkout develop` 또는 신규 피처 브랜치 전환 승인을 즉시 유도합니다.
+ 0. **세션 시작 시 브랜치 자가 진단 및 피처 분기**:
+    - 에이전트는 작업을 시작할 때 항상 현재 브랜치 상태를 식별합니다.
+    - `main` 브랜치뿐만 아니라 `develop` 브랜치에서도 직접 코드 수정 작업을 진행하는 것을 엄격히 금지합니다.
+    - 현재 브랜치가 `main` 또는 `develop`인 경우, 소스 코드 수정 작업을 개시하기 전에 반드시 `develop` 브랜치를 기준으로 신규 작업 브랜치(`feature/###-<name>` 또는 `hotfix/###-<name>`)를 생성하고 전환하는 명령어를 사용자에게 채팅으로 제시하여 실행 승인을 얻어야 합니다.
```
