# 080-enforce-feature-branch.review.md

이 검토서는 `develop` 브랜치 직접 수정 방지 규칙 도입 전과 후를 검토합니다.

## 🛠️ 변경 대비표

| 구분 | 변경 전 | 변경 후 (적용 내용) |
| :--- | :--- | :--- |
| **`develop` 직접 작업 제한** | 금지하지 않아 에이전트가 `develop` 브랜치에 직접 코드를 작성하고 커밋함 | `develop` 브랜치 수정도 엄격히 금지하여, 항상 독립된 피처/버그 브랜치(`feature/*`, `hotfix/*`)에서 개발을 수행하도록 표준화 |
| **세션 진입 시 행동 수칙** | `main` 브랜치인 경우에만 사용자에게 경고하고 `develop`으로 이동을 제안함 | `main` 및 `develop` 브랜치인 경우 모두, 실제 작업을 시작하기 전에 `git checkout -b feature/###-<name>` 명령을 생성하여 명시적 승인을 받도록 의무화 |

## 🔍 변경점 요약 (`.agents/rules/git_flow.md`)
```diff
- 0. **세션 시작 시 브랜치 자가 진단**:
-    - 에이전트는 작업을 시작할 때 항상 현재 브랜치가 `main` 브랜치인지 파악합니다.
-    - `main` 브랜일 경우 코드 수정을 절대 진행하지 않고 사용자에게 경고한 후 `git checkout develop` 또는 신규 피처 브랜치 전환 승인을 즉시 유도합니다.
+ 0. **세션 시작 시 브랜치 자가 진단 및 피처 분기**:
+    - 에이전트는 작업을 시작할 때 항상 현재 브랜치 상태를 식별합니다.
+    - `main` 브랜치뿐만 아니라 `develop` 브랜치에서도 직접 코드 수정 작업을 진행하는 것을 엄격히 금지합니다.
+    - 현재 브랜치가 `main` 또는 `develop`인 경우, 소스 코드 수정 작업을 개시하기 전에 반드시 `develop` 브랜치를 기준으로 신규 작업 브랜치(`feature/###-<name>` 또는 `hotfix/###-<name>`)를 생성하고 전환하는 명령어를 사용자에게 채팅으로 제시하여 실행 승인을 얻어야 합니다.
```
