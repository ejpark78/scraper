# 📋 작업 계획서: Git Flow 브랜치 사전 검사 규칙 추가 (074-git-flow-branch-check-rule.plan.md)

이 문서는 에이전트가 작업 착수 시 현재 브랜치가 `main`인지를 선제 검사하여 잘못된 직접 커밋/푸시를 예방하도록 글로벌 및 프로젝트 규칙 문서(AGENTS.md 및 git_flow.md)를 업데이트하는 계획을 수립합니다.

---

## 1. 🔍 문제 상황 및 필요성
- 이전 세션에서 작업 중 `main` 브랜치에 직접 커밋이 작성되고 `push`가 실행되어 Git Flow 표준 전략이 깨지는 문제가 있었습니다.
- 에이전트 행동 지침(AGENTS.md) 및 세부 가이드라인(git_flow.md)에 **"작업 착수 시 브랜치 선제 확인 의무"**를 강력한 제약 사항으로 명시하여, 향후 동일한 실수가 반복되는 것을 방지합니다.

---

## 2. 🛠️ 해결 방안

### 개선 1: `apps/viewer/AGENTS.md` (혹은 프로젝트 루트 `AGENTS.md`) 업데이트
- **`Git Flow 브랜치 전략 및 에이전트 행동 지침`** 섹션 아래에 다음 하위 규칙을 추가합니다:
  - "작업 세션 시작 시 반드시 현재 git 브랜치를 식별하고, `main` 브랜치일 경우 코드 수정을 시작하기 전에 `git checkout develop` 또는 `git checkout -b feature/*` 전환을 제안할 것."

### 개선 2: `.agents/rules/git_flow.md` 가이드라인 보완
- **`3. 작업 및 병합(Merge) 절차`** 섹션의 첫 단계에 브랜치 자가 진단 및 전환 단계를 의무화하는 규칙을 상세히 추가합니다.

---

3. 작업 변경 이력을 커밋하고 완료 보고서 작성.
